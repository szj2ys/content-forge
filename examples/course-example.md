# 案例：用 ContentForge 设计一节编程课

> 本案例完整记录了一节在线编程课的设计过程，展示 ContentForge 在课程场景下的应用。

**课程主题**：为什么你的 Git 分支总是一团乱？
**目标学员**：有 1-3 年经验的开发者，在团队协作中使用 Git 但经常遇到分支混乱的问题
**课程时长**：12 分钟
**最终耗时**：2.5 小时（选题论证 0.5h → 单节设计 1h → 不平庸标准检查 0.5h → 修改打磨 0.5h）

---

## Step 1: 选题论证

### 为什么选这个题目

Git 课程市面上不缺，但大多数课程的切入点是"Git 命令怎么用"——`git branch`、`git merge`、`git rebase` 的操作教学。这类内容搜一搜文档就能解决。

真正让开发者头疼的不是命令，而是**团队协作中的分支策略**：该用什么分支模型？谁来决定何时合并？为什么每次上线前分支冲突一堆？

这些问题命令教学回答不了。我要从"团队协作"角度讲分支管理，而不是从"Git 命令"角度讲。

### 评分卡打分

| 维度 | 分数（1-5） | 理由 |
|------|:-----------:|------|
| 需求强度 | 5 | 几乎每个使用 Git 的团队都会遇到分支混乱的问题，这是日常痛点 |
| 独特价值 | 3 | "从协作角度讲分支"的视角不算全新，但系统讲清楚的不多 |
| 能力匹配 | 5 | 在 3 个团队主导过分支策略的制定和迁移，有一手实践经验 |
| 竞品空间 | 3 | 竞品多但质量参差，大部分停留在命令教学层面，有差异化空间 |
| 传播潜力 | 3 | Git 话题天然有受众基础，但不算"出圈"型选题 |
| **总分** | **19** | |

19 分，属于"值得做"的范围（15 分以上）。不是爆款型选题，但痛点明确、差异化清晰。

### 竞品分析

调研了 B站、YouTube、掘金上排名前 10 的 Git 分支相关内容：

| 竞品类型 | 数量 | 典型内容 | 缺什么 |
|----------|:----:|---------|--------|
| 命令教学型 | 6 | "git branch 完全指南" | 只讲命令不讲策略，学完还是不知道该用哪种分支模型 |
| 理论介绍型 | 3 | "Git Flow vs GitHub Flow" | 介绍了模型但不讲如何选择，也没讲适用条件 |
| 实战经验型 | 1 | "我们团队如何管理分支" | 个案分享，缺乏通用框架 |

**差异化定位**：不教命令，不罗列模型，而是帮学员回答一个问题——"我的团队应该选哪种分支模型，为什么？"

---

## Step 2: 单节设计

### 基本信息

使用 [lesson-design 模板](../templates/course/02-lesson-design.md) 填写。

- **标题**：为什么你的 Git 分支总是一团乱？
- **学员画像**：1-3 年经验的开发者，日常用 Git 但没有系统学过分支策略
- **学完能做到什么**：能说出 3 种主流分支模型的适用场景，并为自己的团队选择合适的模型
- **时长**：12 分钟

### 知识点拆分

只讲 3 个知识点，不贪多：

| # | 知识点 | 时长 | 核心输出 |
|---|--------|:----:|---------|
| 1 | 常见分支模型对比 | 4 min | 一张对比表：Git Flow / GitHub Flow / Trunk-based 的适用场景 |
| 2 | 如何选择适合你团队的模型 | 4 min | 一个决策树：根据团队规模、发布频率、CI 成熟度选择模型 |
| 3 | 5 个让分支保持干净的习惯 | 4 min | 一份可直接执行的习惯清单 |

### 开场钩子

> 周五下午 3 点，你的团队准备上线。你打开 Git 仓库，发现 `develop` 分支上有 12 个未合并的 feature 分支，其中 3 个已经过期但没人删，2 个互相冲突。你花了整整两个小时解决冲突，最后上线推迟到了晚上 8 点。
>
> 问题出在哪？不是因为你的团队不会用 Git 命令——每个人都会 `git merge`。问题在于：你们从来没有认真讨论过"分支该怎么管"。

### 内容大纲

```
一、三种分支模型，各适合什么团队？（4 min）
   - Git Flow：适合发布节奏慢、需要维护多个版本的项目（如桌面软件、SDK）
   - GitHub Flow：适合持续部署、只维护一个生产版本的 Web 项目
   - Trunk-based Development：适合 CI/CD 成熟、团队协作紧密的团队
   → 关键信息：Git Flow 在大多数 Web 团队是过度设计（反直觉点）

二、你的团队该选哪个？一棵决策树搞定（4 min）
   - 问题1：你的项目需要同时维护多个版本吗？→ 是 → Git Flow
   - 问题2：你的团队有成熟的 CI/CD 和自动化测试吗？→ 是 → Trunk-based
   - 问题3：以上都不是？→ GitHub Flow（最安全的起步选择）
   → 展示 Google 的实践：全公司用 Trunk-based，但前提是超强的 CI 基础设施

三、5 个让分支保持干净的习惯（4 min）
   1. 分支命名规范：type/ticket-description（如 feat/PROJ-123-add-login）
   2. 单一职责：一个分支只做一件事，PR 不超过 400 行
   3. 短命分支：feature 分支不超过 3 天，超过就拆分
   4. 及时清理：合并后立即删除分支，不留尸体
   5. 主干保护：main/master 永远可部署，用 PR + CI 守门
   → 金句："管理分支的本质不是管理代码，是管理协作节奏"
```

### 结尾设计

不做"总结回顾"式的走过场收尾。直接给行动指引：

> 今天做一件事：打开你团队的 Git 仓库，数一下现在有多少个超过一周没更新的分支。如果超过 5 个，说明你们的分支策略需要认真聊一聊了。

## Step 3: 不平庸标准检查

逐条对照 [不平庸的5条标准](../methodology/00-not-mediocre.md)：

### 1. 有信息增量 ✅ 达标

- "Trunk-based Development 的适用条件"是大多数初中级开发者不知道的。很多人听过这个概念，但不清楚什么条件下才能用、为什么 Google 能用但自己的团队用不了。
- "决策树"是一个实用的思维工具，市面上的竞品没有提供类似的选择框架。

### 2. 有反直觉观点 ✅ 达标

- **"Git Flow 在大多数 Web 团队是过度设计"** — 这是一个有力的反直觉点。很多团队默认选 Git Flow，以为它是"最佳实践"，实际上它是为特定场景设计的（需要维护多版本的传统软件），大部分 Web 项目用不到那么复杂的分支结构。
- 这个观点有数据支撑：Git Flow 原作者 Vincent Driessen 自己在 2020 年也补了一条说明，建议 Web 项目考虑更简单的模型。

### 3. 有真实证据 ⚠️ 需要补强

**现状**：目前只有泛泛的"Google 用 Trunk-based"的提法，缺乏具体数据。

**改进方案**：
- 补充 Google 的具体实践：单一代码仓库（monorepo）+ 超过 30,000 名工程师 + 每天约 70,000 次提交，全部在主干上完成。来源：Google Engineering Practices 文档。
- 补充一个反面案例：某团队从 Git Flow 迁移到 GitHub Flow 后，发布周期从 2 周缩短到 1 天，合并冲突减少 70%。
- 补充 Git Flow 原作者 Vincent Driessen 2020 年的补充说明作为权威引用。

**改进后状态**：✅

### 4. 读完能行动 ✅ 达标

学员学完后有明确的行动路径：
- 用决策树为自己的团队选择分支模型（立即可做）
- 用 5 个习惯清单逐项检查团队现状（立即可做）
- 结尾的"今天做一件事"给了最小行动步骤

### 5. 有分享冲动 ⚠️ 需要补强

**现状**：内容实用但"金句密度"不够。学员觉得有用，但不一定会想转发。

**改进方案**：
- 加入金句：**"管理分支的本质不是管理代码，是管理协作节奏"** — 把技术问题升维到团队管理层面，容易引发共鸣和转发
- 开场钩子的场景描写要更有画面感，让学员看完就想说"这说的就是我们团队"
- 决策树做成一张可保存的图片，方便学员截图分享到团队群

**改进后状态**：✅

### 检查结果汇总

| # | 标准 | 初稿状态 | 改进后 |
|---|------|:--------:|:------:|
| 1 | 有信息增量 | ✅ | ✅ |
| 2 | 有反直觉观点 | ✅ | ✅ |
| 3 | 有真实证据 | ⚠️ | ✅ 补了 Google 数据 + 迁移案例 + 原作者引用 |
| 4 | 读完能行动 | ✅ | ✅ |
| 5 | 有分享冲动 | ⚠️ | ✅ 加了金句 + 可保存的决策树图片 |

## Step 4: 复盘

### 时间投入

| 环节 | 耗时 | 占比 |
|------|:----:|:----:|
| 选题论证（评分卡 + 竞品分析） | 0.5h | 20% |
| 单节设计（知识点拆分 + 大纲 + 开场钩子） | 1h | 40% |
| 不平庸标准检查 + 修改 | 0.5h | 20% |
| 最终打磨（金句、数据补充、结尾调整） | 0.5h | 20% |
| **合计** | **2.5h** | 100% |

平时设计一节课大约 1.5 小时。这次多了 1 小时，主要花在选题论证和不平庸标准检查上。

### 最大收获

不平庸标准检查让我发现"真实证据"严重不足。如果没有这一步，我大概率会用"Google 用的就是 Trunk-based"这种模糊说法一笔带过。补了具体数据和迁移案例后，内容的说服力明显提升——从"我觉得 Trunk-based 好"变成了"这是 Google 30,000 名工程师验证过的，而且有团队迁移后发布周期缩短了 14 倍"。

### 最大意外

原来打算讲 4 个知识点，第 4 个是"如何制定分支命名规范"。但在不平庸标准检查时发现，4 个知识点塞在 12 分钟里太赶，每个点只能浅尝辄止，违反了"有信息增量"的标准。砍到 3 个后，每个知识点有 4 分钟的充裕时间，可以把案例和数据讲透。

分支命名规范没有丢掉，而是降级成了"5 个习惯"中的第一条——一句话讲清规则就够了，不需要单独占一个知识点。

### 对比：有 ContentForge vs 没有

| 维度 | 没有 ContentForge | 有 ContentForge |
|------|------------------|----------------|
| 选题 | 凭感觉选，选了就干 | 评分卡打分，竞品分析明确差异化 |
| 知识点 | 想到多少讲多少，容易贪多 | 标准检查后主动砍掉 1 个，聚焦 3 个 |
| 证据 | "Google 也这么干"一笔带过 | 补了具体数据、迁移案例、权威引用 |
| 金句 | 没有刻意设计 | 标准检查后专门加了 1 句，提升分享冲动 |
| 耗时 | 1.5h | 2.5h（多 1h） |
| 质量 | 能用，但平庸 | 有差异化、有证据、有记忆点 |

多花的 1 小时，换来的是从"一节普通的 Git 课"到"一节学员会截图分享到团队群的课"的差距。

---

**核心体会**：不平庸标准不是用来"打分交作业"的，而是用来发现盲区的。这次最有价值的两个发现——证据不足和金句缺失——都是自己写的时候完全没意识到的问题。
