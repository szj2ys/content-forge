# 审核阶段检查清单

> 自己写的东西自己看不到问题，这是认知规律而非个人缺陷。审核阶段的价值在于引入"他人视角"——包括系统化的自检、同行的新鲜眼光、以及不可触碰的红线排查。

---

## 自检清单

在邀请他人评审之前，先完成自检，不要把本该自己发现的问题浪费在评审人身上。

- [ ] 重读选题定位：「本文帮助 [谁] 解决 [什么问题]，通过 [什么角度]」——成品是否兑现了这个承诺？
- [ ] 核心论点测试：如果读者只记住一句话，会是哪句？这句话是否足够清晰有力？
- [ ] "所以呢"测试：读完全文，读者下一步会做什么？如果答案是"没什么"，说明行动号召不够
- [ ] 标题与内容匹配度：标题承诺的，正文是否完全交付？没有标题党，也没有"货不对板"
- [ ] 原创性自查：所有引用是否标注来源？是否有无意中与他人内容过于相似的段落？
- [ ] 错别字和语法检查：使用工具辅助检查一遍（不要完全依赖人眼）
- [ ] 链接全部可点击且指向正确页面

## 同行评审清单

找 1-2 位了解目标读者的人进行评审。提供评审框架，避免收到"挺好的"这种无效反馈。

### 给评审人的引导问题

请评审人针对以下维度给出反馈，每条标注为：**必须改** / **建议改** / **亮点**

- [ ] 开头是否吸引你继续读下去？在哪里产生了"想关掉"的冲动？
- [ ] 逻辑链条是否通顺？哪里觉得"跳跃了"或"没说服我"？
- [ ] 案例和数据是否可信？哪个最有说服力？哪个最弱？
- [ ] 是否有你觉得多余的部分？（指出具体段落）
- [ ] 是否有你觉得缺失的部分？（读完后还想知道什么）
- [ ] 整体阅读体验打分（1-10）并说明理由
- [ ] 你会把这篇文章分享给谁？为什么？

### 评审结果处理

- [ ] 收集所有反馈，分类整理（必须改 / 建议改 / 亮点）
- [ ] "必须改"全部处理完毕
- [ ] "建议改"逐条评估，采纳或写明不采纳理由
- [ ] 评审人确认修改后的版本已解决提出的"必须改"问题
- [ ] 如果两位评审人意见冲突，以目标读者的需求为准做取舍

## 红线检查清单

无论内容多好，触碰红线则不能发布。这些项目没有灰色地带。

- [ ] **事实准确性**：所有数据、日期、人名、公司名是否经过核实？
- [ ] **版权合规**：引用图片、数据、文字是否有授权或符合合理使用？
- [ ] **隐私保护**：是否涉及未经授权的个人信息、内部数据、商业机密？
- [ ] **立场风险**：是否包含可能引起争议的价值判断？如有，是否已充分考虑不同视角？
- [ ] **利益冲突**：是否涉及利益相关方？如有，是否已透明披露？
- [ ] **技术安全**：代码示例是否安全？是否可能导致读者的系统出问题？
- [ ] **时效性**：引用的信息是否仍然有效？（技术文章特别注意版本号和 API 变化）

## 终审决定

- [ ] 自检全部通过
- [ ] 同行评审"必须改"全部解决
- [ ] 红线检查零问题
- [ ] 做出终审决定（三选一）：
  - **通过** → 进入发布阶段
  - **有条件通过** → 列出剩余修改项，修改后无需重新评审
  - **打回** → 需要重大修改，修改后需重新评审

---

**通过标准**：终审决定为"通过"或"有条件通过"（且条件已满足）。

**常见失败模式**：
- 跳过自检直接请人评审（浪费评审人时间和善意）
- 只找"会说好话的人"评审（评审的价值在于发现问题，不是获得肯定）
- 红线检查走过场（"应该没问题"不等于"确认没问题"）
- 对"必须改"的反馈消极抵触（评审不是对你的否定，是对内容的打磨）
